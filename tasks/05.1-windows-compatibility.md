# 任务 05.1：Windows 兼容性支持

## 目标

使 Lattice 能够在 Windows 平台上正常运行，解决 LocalSandbox 硬编码 `sh -c` 导致的跨平台问题，并提供 Windows 用户文档和脚本支持。

## 分支

`feat/windows-compatibility`

## 依赖

- 任务 5（LocalSandbox）✅
- 任务 15（工具系统）✅

## 关联 Issue

- [#45 Windows compatibility support](https://github.com/hhllhhyyds/Lattice/issues/45)

## 背景

### 当前问题

1. **硬编码 Unix shell**：`LocalSandbox` 在 `local_sandbox.rs:86-87` 硬编码使用 `sh -c` 执行命令
   ```rust
   let mut cmd = Command::new("sh");
   cmd.args(["-c", cmd_str]);
   ```

2. **Windows 不兼容**：Windows 原生没有 `sh` 命令（除非安装 Git Bash 或 WSL）

3. **工具调用失败**：所有通过 `BashTool` 执行的命令在 Windows 上都会失败

4. **文档缺失**：README.md 只提供 Unix 风格的命令示例，缺少 Windows 说明

5. **脚本不可用**：`scripts/check.sh` 和 `scripts/test-local.sh` 在 Windows 上无法直接运行

### 影响范围

- `crates/sandbox-local/src/local_sandbox.rs` - 核心执行逻辑
- `crates/tools/src/bash.rs` - BashTool 描述（可能需要更新文档字符串）
- `README.md` - 缺少 Windows 构建和运行说明
- `docs/ARCHITECTURE.md` - Sandbox 章节需要说明跨平台行为
- `scripts/` - 缺少 Windows 版本的脚本

### 阶段 3：TDD Phase 2 — 实现平台检测逻辑

**文件**：`crates/sandbox-local/src/local_sandbox.rs`

**实现策略**：
1. 使用 Rust 的条件编译在编译时选择 shell
2. Unix/Linux/macOS: `sh -c`
3. Windows: `cmd.exe /C`

**代码修改**：

```rust
#[async_trait]
impl Sandbox for LocalSandbox {
    #[instrument(skip(self))]
    async fn execute(
        &self,
        _tool: &str,
        params: serde_json::Value,
    ) -> Result<ExecutionResult, SandboxError> {
        let cmd_str = params
            .get("command")
            .and_then(|v| v.as_str())
            .ok_or_else(|| {
                SandboxError::ExecutionFailed("missing 'command' in params".to_string())
            })?;

        // Platform-specific shell selection
        #[cfg(unix)]
        let mut cmd = {
            let mut c = Command::new("sh");
            c.args(["-c", cmd_str]);
            c
        };

        #[cfg(windows)]
        let mut cmd = {
            let mut c = Command::new("cmd");
            c.args(["/C", cmd_str]);
            c
        };

        if let Some(ref dir) = self.work_dir {
            cmd.current_dir(dir);
        }

        let result =
            timeout(self.timeout, cmd.output())
                .await
                .map_err(|_| SandboxError::Timeout {
                    timeout_secs: self.timeout.as_secs(),
                })?;

        match result {
            Ok(output) => Ok(ExecutionResult {
                stdout: String::from_utf8_lossy(&output.stdout).to_string(),
                stderr: String::from_utf8_lossy(&output.stderr).to_string(),
                exit_code: output.status.code().unwrap_or(-1),
            }),
            Err(e) => Err(SandboxError::ExecutionFailed(e.to_string())),
        }
    }
}
```

```markdown
### 3. Sandbox（沙箱）—— Agent 的双手

**定义**：隔离的工具执行环境，按需创建，可替换。

**关键特性**：
- **隔离**：沙箱内的代码无法访问框架内部或凭据
- **可替换**：崩溃了换一个新的，ControlLoop 视之为一次工具调用失败

- **跨平台**：LocalSandbox 根据操作系统自动选择合适的 shell
  - Unix/Linux/macOS: `sh -c`
  - Windows: `cmd.exe /C`
```

```markdown
# lattice-sandbox-local

## Purpose

Local subprocess-based sandbox implementation. Executes commands as local subprocesses with no isolation (development/testing only).

## Key Types

- `LocalSandbox` — executes commands via `tokio::process::Command`

## Platform Support

LocalSandbox automatically selects the appropriate shell based on the target platform:

- **Unix/Linux/macOS**: Uses `sh -c <command>`
- **Windows**: Uses `cmd.exe /C <command>`

The shell selection happens at compile time using `#[cfg(unix)]` and `#[cfg(windows)]`.

## Security Warning

**This provides no isolation.** Commands run with the same OS user privileges as the Lattice process. Do not use in untrusted environments.

## Dependencies

- Depends on: `lattice-core`
- Depended on by: `lattice-tools` (BashTool)
```

```powershell
#!/usr/bin/env pwsh
# Windows equivalent of test-local.sh
# Requires .env file with LATTICE_API_KEY

if (-not (Test-Path .env)) {
    Write-Host "Error: .env file not found" -ForegroundColor Red
    Write-Host "Create .env with: LATTICE_API_KEY=your-key" -ForegroundColor Yellow
    exit 1
}

Write-Host "Loading .env..." -ForegroundColor Cyan
Get-Content .env | ForEach-Object {
    if ($_ -match '^([^=]+)=(.*)$') {
        $name = $matches[1]
        $value = $matches[2]
        Set-Item -Path "env:$name" -Value $value
    }
}

Write-Host "Running integration tests with real LLM..." -ForegroundColor Cyan
cargo test --workspace -- --ignored

if ($LASTEXITCODE -eq 0) {
    Write-Host "Integration tests passed!" -ForegroundColor Green
} else {
    Write-Host "Integration tests failed!" -ForegroundColor Red
    exit $LASTEXITCODE
}
```

## 注意事项

1. **不改变 BashTool 的名称**：虽然在 Windows 上使用 `cmd.exe`，但保持 `BashTool` 名称以维持 API 兼容性
2. **shell 命令兼容性**：用户需要确保命令在目标平台的 shell 中有效（如 `ls` vs `dir`）
3. **路径分隔符**：Rust 的 `std::path` 会自动处理，无需特殊处理
4. **换行符**：Git 会自动处理 CRLF/LF 转换
